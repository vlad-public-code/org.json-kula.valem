# Valem Documentation

Deterministic reactive computation runtime for AI-generated structured data models.
This is the documentation map. Find your audience below and start there.

**▶ [Try the live sandbox](https://valem.onrender.com/)** — a zero-setup public demo: describe a
domain in plain language, watch an LLM generate a ModelSpec, then mutate fields and see derivations,
constraints, and effects react live.

> `architecture/` document how the shipped system works; `guides/` are task-oriented how-tos.

## What is Valem?

Valem runs a declarative **model spec** as a live, spreadsheet-like reactive computation over a
JSON document. You hand it a [`ModelSpec`](reference/model-spec-format.md) — a schema plus JSONata
**derivations** (computed fields), **constraints** (invariants), **effects** (guarded I/O), and an
optional UI **view** — and it executes it: mutate a base field, and every dependent value
recomputes in dependency order, invariants are enforced, effects fire and fold their results back,
and every step is traced and auditable. Execution is deterministic and reproducible: replaying the
same mutations reproduces the same state, and replay never re-runs external I/O.

A spec is usually **generated by an LLM** from a plain-text domain description (see
[generating-specs-with-llm.md](guides/generating-specs-with-llm.md)), then compiled and validated
by the runtime.

### What it's for

- **The compute-and-validate layer for LLM / agent systems** that need durable, structured world
  state instead of re-deriving everything in the prompt — drive a model over
  [REST](reference/api-reference.md), the [console](reference/api-reference.md#3-console-json-protocol),
  or [MCP](guides/mcp-server.md).
- **Quoting, eligibility, and calculator tools** over structured data (insurance quotes, benefits
  eligibility, loan amortization) — the recurring "wire the same compute-and-validate layer per
  project" work, expressed once as a spec. See the [examples gallery](guides/examples-gallery.md).
- **Stateful agents, human-in-the-loop workflows, and time-driven systems** — effects let the
  deterministic core reach the world (HTTP, an LLM call, a scheduled timer) while staying auditable
  and replayable.

### Why a validated spec is safe to run

A `ModelSpec` is **not general-purpose code**. All of its pure logic — every derivation,
constraint, meta-derivation, and default — is written in **JSONata**, a sandboxed expression
language with no file, process, or arbitrary-network access: an expression can only compute over
the model's own JSON. The *only* way a spec reaches outside itself is through **effects**, and those
run through a small, fully-controlled set of executors (`caller`/`server`/`llm`/`timer` plus any
operator-installed plugin) that the deployment governs: the `server` executor is fronted by an SSRF
**egress guard** (loopback/private-IP blocking, scheme restriction, optional host allowlist and
response-size cap), and `valem.effects.kinds.enabled` lets an operator restrict which effect kinds
may run at all. So once `ModelSpecValidator` accepts a spec, running it cannot execute arbitrary
code or reach arbitrary hosts — it can only compute over its own state and perform the specific,
guarded I/O the operator has enabled. See [security-model.md](reference/security-model.md).
